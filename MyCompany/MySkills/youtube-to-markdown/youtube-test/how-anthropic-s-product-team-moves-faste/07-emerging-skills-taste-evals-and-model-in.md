# Emerging Skills: Taste, Evals, and Model Intuition

> The hardest skill is defining what the product should look like a month from now under model-capability uncertainty. Cat's playbook: use the model heavily, ask it to introspect, build a small set of trusted reviewers, write ~10 great evals, and audit the system prompt at every model release.

## Key Points
- **The hardest skill** is being able to "define what the product should look like a month from now" under deep uncertainty about model capability shifts and user behavior change.
- "It is very hard to be the right amount of **AGI-pilled**" — too pilled and you build for a model that isn't here yet; too unpilled and you over-engineer crutches the next model will eat for breakfast.
- **Build product taste through model usage**: ask the model to **introspect on its own behaviors**. When it does something unexpected (e.g., runs tests but doesn't actually use the UI), ask why. Often the model itself reveals the harness flaw — confusing system prompt, missed verification step, sub-agent skipped its work.
- Find a **handful of trusted "taste" reviewers** (Cat names ~5 people). Two reviewers Cat singles out: **Amanda**, who shapes Claude's character ("incredibly hard role; the task is so ambiguous; even coding is easier because you can verify success"), and **the Cloud Code team**, polled at lunch when a new model is in test.
- Feedback at team lunches sounds like: "this model isn't fully explaining its thinking, it's too abrupt"; "loves writing memories but quality is unclear"; "loves to test itself" — used as **hypotheses** that the team then tests against data.
- **10 great evals beat hundreds of mediocre evals.** "You don't need to build hundreds of evals for them to be useful. Just building 10 great evals is important for helping the team quantify what the goal is and what their progress towards it is."
- Cat personally writes evals for features that need product definition: the output is "5 evals, here's how to run them, these succeed, these don't, this is the prompt that increases the success rate."
- **Claude's character (low-ego, positive, bias-to-action, earnest feedback) is core product strategy** — not flavor. People love Claude's "vibe" the way they love a coworker; it's why Open-Claud users were upset about losing their preferred personality.
- **Each new model release triggers a system-prompt audit.** The team reads through every section and asks: "Does the model still need this reminder?" If not, delete it.
- **The classic crutch example: the to-do list.** Early Cloud Code couldn't finish 20-call-site refactors; Sid added a to-do tool to force completion; Opus 4 and later use it naturally without prompting. Today the to-do list is a UI affordance, not a model crutch.
- **Build for the next model, not today's**: code review failed multiple times until Opus 4.5/4.6 + Sonnet 4.6 finally crossed the reliability bar. Anthropic's engineers now rely on it as a merge gate.
- "It's pretty important to build products that don't necessarily work yet so that you know what is missing. Then with the newest model, you swap it in to the prototype you've already made."

## Details

**Why "define the product a month from now" is the bottleneck skill.** The PM has to reason about (1) what models will be capable of one month out, and (2) how user behavior will shift in response. Cat: "There's a lot of ambiguity... but there are patterns the best PMs can see based on how users are abusing the limits of the existing product. The best PMs can sense that, can set a direction, and can steadily execute towards it — and change the path if model capabilities are much better or worse than expected."

**The AGI-pilled tightrope.** Cat: "It is very hard to be the right amount of AGI-pilled. Everyone can see this future where models are extremely smart and can do almost everything, in which case you actually don't need that complicated a product. You can have a text box again where you tell the model what you want and it can add any tool or integration to get the job done. It's kind of very easy to build the product for the super-AGI strong model. The hard thing is figuring out for the **current** model — how do you elicit the maximum capability? How do you guide users to interact with the model's strengths and patch its weaknesses?"

**Three concrete ways to build the skill.**

1. **Ask the model to introspect.** "Sometimes when I notice the model does something unexpected — for example, the model will make a front-end change and run tests but not actually use the UI — it's actually pretty useful to ask the model to reflect on why it did this. Sometimes they'll say there was something confusing in the system prompt, or I didn't realize the front-end verification was part of this task, or I delegated the verification to a sub-agent and the sub-agent didn't do the test and I didn't check its work." This reveals harness gaps.

2. **Cultivate a small trusted taste circle.** "Usually there's a handful of people who are much better than others at articulating what makes a specific model or model-harness combination good. Not everyone's feedback is as qualified. Finding a group of those like five people you trust is really important for getting very fast feedback." Cat names two: **Amanda** for Claude's character ("she has an incredible ability not only to mold the character but to articulate what the goals are, what's successful and what's not"); and **the Cloud Code team** broadly, polled at team lunches whenever a new model is in test.

3. **Write ~10 great evals.** Cat: "Eval is this underappreciated thing that more PMs and more engineers should be working on." Important features (memory is her example) benefit a lot; not every feature needs them. Her personal pattern when she builds evals: "5 evals I made, here's how to run them, these succeed, these don't, here's the prompt that increases the success rate." Output is portable and team-readable.

**Character is product, not garnish.** When Lenny mentions Ben Mann's framing — "the personality is what makes Claude so good at so many things" — Cat doubles down. Claude is "lighthearted and fun" but also "extremely competent at your task." Users love that Claude is **low-ego** ("oh shoot, thanks for telling me, let me fix it"), **positive in the face of insurmountable tasks**, and gives **earnest feedback** rather than agreeing with every input. "We try to imbue this into Claude because we think it makes it a lot more enjoyable to work with." This is why OpenClaud users specifically named the lost personality as their main grievance.

**The system-prompt audit ritual.** Every model release: the team reads through the entire system prompt and asks, for each section, "does the model really need this reminder anymore?" If not, delete it. Most additions to a Claude product are crutches that compensate for current-model weaknesses.

**The to-do list, in full.** When Cloud Code launched, asking it for a 20-call-site refactor would change five sites and stop. Sid on the team modeled what a human would do — "make a list of everything they need to change, similar to how in VS Code you would look up all the call sites... and you would go through them one by one." So he added a to-do list tool. With the to-do, Claude finished the 20 changes. Earlier models had to be reminded to finish the list. **Opus 4 and later just use it naturally.** "These days, the to-do list is still nice to have as a user because then you can more clearly see what Claude is working on. But honestly, it's such a deemphasized part of the product right now... it's really not necessary for it to make thorough changes anymore."

**Code review as the build-for-next-model exemplar.** Anthropic tried to ship code review multiple times — including the early `/code-review` slash command. None of those were good enough that engineers would rely on them. Only with Opus 4.5/4.6 and Sonnet 4.6 did the bar get crossed: "we are now able to run multiple code-review agents simultaneously to traverse the entirety of the codebase and synthesize a set of real issues an engineer needs to address before merge." Cloud Code's own engineering team now uses code review as a merge gate. Cat's general advice: "Build products that don't necessarily work yet so that you know what is missing for this product to work. Then with the newest model, you can just swap it in to the prototype you've already made and see — does this new model close that gap."
